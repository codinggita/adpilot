SECTION 1 — EXECUTIVE SUMMARY
================================================================================

AdPilot AI is a fully functional MERN-stack SaaS product that connects to small
brand Google Ads accounts, detects wasted ad spend using AI, and applies
automated optimizations — all at zero infrastructure cost using free tiers.

The product addresses a clear, scored problem:
  Severity Score:    5 / 5
  TAM Score:         50
  Whitespace Score:  5 / 5
  Frequency Score:   7.5
  Itch Score:        85

Target user: Small brand owners and solo marketers running Google Ads with
monthly budgets between ₹10,000 and ₹5,00,000 (approx. $120–$6,000 USD).
They lack the time, expertise, and budget to use enterprise tools like
Optmyzr, WordStream, or Google's own Smart Campaigns effectively.

Core value proposition:
  "Connect your Google Ads account in 30 seconds. AI finds your wasted spend,
   fixes it automatically, and shows you the results — no expertise required."

SECTION 2 — PROBLEM STATEMENT & MARKET CONTEXT
================================================================================

CORE PROBLEM:
Small brands lose advertising spend because they lack the expertise and time
to continuously optimize campaigns, making manual keyword tuning and budget
allocation inefficient for low-budget Google Ads accounts.

Specific pain points:
  1. Broad match keywords drain budgets on zero-intent search queries.
  2. Bid strategies are set once and never revisited.
  3. Budget is distributed evenly across campaigns regardless of performance.
  4. Mobile, geographic, and time-of-day bid adjustments are ignored.
  5. Negative keyword lists are empty or stale.
  6. Ad quality scores degrade over time without copy refresh.
  7. No one is watching the account daily — small brands check weekly at best.

MARKET GAP:
  - Enterprise tools (Optmyzr, WordStream): $299–$999/month. Too expensive.
  - Google Smart Campaigns: Black box. No transparency. Poor for niche products.
  - Manual management: Requires 10+ hours/week. Small brands cannot afford it.
  - AdPilot fills: AI-powered, transparent, affordable (₹0 to start), MERN-based.

SECTION 6 — SYSTEM ARCHITECTURE
================================================================================

HIGH-LEVEL ARCHITECTURE:

  [React SPA on Vercel]
         |
         | HTTPS / REST API
         |
  [Express.js API on Render.com]
         |
    _____|_____________________________
    |          |           |          |
[MongoDB    [Google     [Gemini    [Resend
 Atlas]      Ads API]    AI API]    Email]
                |
         [PostHog Analytics]

DATA FLOW FOR AI AUDIT:
  1. User clicks "Run AI Audit" in React frontend
  2. POST /api/audit/run → Express handler
  3. Express fetches campaign + keyword data from Google Ads API
  4. Raw data is structured and sent to Gemini Flash API
  5. Gemini returns JSON with prioritized recommendations
  6. Recommendations are saved to MongoDB (AuditResult collection)
  7. Response sent back to React frontend
  8. Frontend renders recommendation cards in real-time
  9. User clicks "Apply" → POST /api/audit/apply/:recommendationId
  10. Express calls Google Ads API to execute the change
  11. Change is logged in MongoDB (ActionLog collection)
  12. PostHog event: apply_recommendation

DATA FLOW FOR AUTOMATION RULES:
  1. node-cron job runs every day at 3:00 AM IST
  2. Fetches all active rules from MongoDB (AutomationRule collection)
  3. For each rule, fetches relevant campaign data from Google Ads API
  4. Evaluates IF condition against live data
  5. If condition is true → executes THEN action via Google Ads API
  6. Logs result in MongoDB (RuleExecutionLog collection)
  7. If notify=true → sends email via Resend

DATA FLOW FOR WEEKLY REPORT:
  1. node-cron job runs every Monday at 9:00 AM IST
  2. Fetches 7-day performance data per account from Google Ads API
  3. Sends structured data to Gemini for narrative generation
  4. Builds HTML email with charts and AI narrative
  5. Generates PDF via jsPDF
  6. Uploads PDF to Cloudinary
  7. Sends email via Resend with PDF link
  8. Saves report metadata in MongoDB (Report collection)

SECTION 8 — API DESIGN (REST ENDPOINTS)
================================================================================

BASE URL: https://api.adpilot.ai/api/v1
All authenticated routes require: Authorization: Bearer <jwt_token>
All responses follow: { success: Boolean, data: Object|Array, error: String }

────────────────────────────────────────────────────────────────────────────────
AUTH ROUTES — /api/v1/auth
────────────────────────────────────────────────────────────────────────────────

POST   /auth/register
  Body: { email, password, name }
  Returns: { user, token }
  Validation: email (valid format), password (min 8 chars), name (min 2 chars)

POST   /auth/login
  Body: { email, password }
  Returns: { user, token }

POST   /auth/logout
  Returns: { message: 'Logged out' }

GET    /auth/google
  Redirects to Google OAuth consent screen
  Scopes: email, profile, https://www.googleapis.com/auth/adwords

GET    /auth/google/callback
  Handles OAuth callback from Google
  Creates/updates user, sets JWT cookie
  Redirects to frontend /dashboard

POST   /auth/refresh
  Body: { refreshToken }
  Returns: { token }

POST   /auth/forgot-password
  Body: { email }
  Returns: { message: 'Reset email sent' }

POST   /auth/reset-password
  Body: { token, newPassword }
  Returns: { message: 'Password reset successful' }

GET    /auth/me
  Returns: { user } (current logged-in user)

────────────────────────────────────────────────────────────────────────────────
GOOGLE ADS ACCOUNT ROUTES — /api/v1/accounts
────────────────────────────────────────────────────────────────────────────────

GET    /accounts
  Returns: [list of connected Google Ads accounts for current user]

POST   /accounts/connect
  Body: { authCode, customerId }
  Exchanges auth code for tokens, stores encrypted in DB
  Returns: { account }

DELETE /accounts/:accountId
  Revokes access and removes account from DB
  Returns: { message: 'Account disconnected' }

GET    /accounts/:accountId/sync
  Triggers manual data sync from Google Ads API
  Returns: { syncJob: { jobId, status, startedAt } }

GET    /accounts/:accountId/sync/status
  Returns: { syncStatus, lastSyncAt, progress }

POST   /accounts/:accountId/permissions
  Body: { canWriteBids, canWriteBudgets, canWriteKeywords }
  Returns: { account }

────────────────────────────────────────────────────────────────────────────────
CAMPAIGNS ROUTES — /api/v1/campaigns
────────────────────────────────────────────────────────────────────────────────

GET    /campaigns
  Query: accountId, status, type, page, limit, sortBy, sortOrder
  Returns: { campaigns: [], total, page, pages }

GET    /campaigns/:campaignId
  Returns: { campaign, recentMetrics: [7 days], topKeywords: [10] }

GET    /campaigns/:campaignId/metrics
  Query: startDate, endDate, granularity (daily|weekly|monthly)
  Returns: { metrics: [] }

GET    /campaigns/:campaignId/keywords
  Query: page, limit, matchType, status, aiTag, sortBy
  Returns: { keywords: [], total }

PATCH  /campaigns/:campaignId/pause
  Calls Google Ads API to pause campaign
  Returns: { campaign }

PATCH  /campaigns/:campaignId/enable
  Calls Google Ads API to enable campaign
  Returns: { campaign }

PATCH  /campaigns/:campaignId/budget
  Body: { dailyBudget }
  Calls Google Ads API to update budget
  Returns: { campaign }

────────────────────────────────────────────────────────────────────────────────
KEYWORDS ROUTES — /api/v1/keywords
────────────────────────────────────────────────────────────────────────────────

GET    /keywords
  Query: accountId, campaignId, adGroupId, aiTag, matchType, page, limit
  Returns: { keywords: [], total }

GET    /keywords/negatives
  Query: accountId, campaignId
  Returns: { negativeKeywords: [] }

POST   /keywords/negatives
  Body: { campaignId, keywords: [{ text, matchType }] }
  Calls Google Ads API to add negative keywords
  Returns: { added: Number, keywords: [] }

PATCH  /keywords/:keywordId/bid
  Body: { bidAmount }
  Calls Google Ads API to update keyword bid
  Returns: { keyword }

PATCH  /keywords/:keywordId/pause
  Calls Google Ads API to pause keyword
  Returns: { keyword }

PATCH  /keywords/:keywordId/enable
  Returns: { keyword }

DELETE /keywords/:keywordId
  Removes keyword from Google Ads (soft delete in DB)
  Returns: { message: 'Keyword removed' }

GET    /keywords/suggestions
  Query: accountId, campaignId
  Uses Gemini AI to suggest new keywords based on existing ones

  - "Remember me" extends refresh token to 30 days

FEATURE 2: GOOGLE ADS ACCOUNT CONNECTION
────────────────────────────────────────────────────────────────────────────────

F2.1 — OAuth2 Connection Flow
  - User clicks "Connect with Google Ads"
  - Redirect to Google OAuth consent screen with adwords scope
  - On callback: exchange code for access + refresh tokens
  - Encrypt both tokens using AES-256 before storing in MongoDB
  - Verify access by making a test call to Google Ads API
  - Store account name, customer ID, currency, timezone

F2.2 — Manual Customer ID Entry
  - Allow users who already authenticated via Google to enter Customer ID
  - Validate format (XXX-XXX-XXXX or numeric)
  - Make test API call to verify access
  - Show error if insufficient permissions

F2.3 — Token Refresh Logic
  - Before every Google Ads API call, check tokenExpiresAt
  - If expired (or within 5 minutes of expiry): call Google token refresh endpoint
  - Update encrypted tokens in MongoDB
  - On refresh failure: mark account syncStatus='error', send alert email

F2.4 — Initial Data Sync
  - After connection, trigger full sync:
    a. Fetch all non-removed campaigns
    b. Fetch all ad groups per campaign
    c. Fetch all keywords per ad group
    d. Fetch last 90 days of daily metrics per campaign
    e. Fetch last 7 days of search term reports
  - Show sync progress to user (polling /sync/status every 3 seconds)
  - Typical sync time: 30–120 seconds depending on account size
  - On complete: redirect user to dashboard with data loaded

F2.5 — Ongoing Sync Schedule
  - Background cron: every 6 hours, sync all active accounts
  - Sync only incremental data (yesterday's metrics + current status)
  - Full sync: every Sunday at 2:00 AM IST
  - Sync errors logged and user notified

FEATURE 3: AI AUDIT ENGINE
────────────────────────────────────────────────────────────────────────────────

F3.1 — Audit Trigger
  - Manual: user clicks "Run AI Audit" button
  - Scheduled: every Monday at 3:00 AM IST (via node-cron)
  - Onboarding: auto-triggered after first successful sync
  - API: POST /audit/run with configuration options

F3.2 — Data Collection Phase (before calling Gemini)
  Collect from MongoDB (already synced data):
  a. All campaigns with last 7-day and 30-day metrics
  b. All keywords with last 7-day and 30-day metrics
  c. Search term reports (what actual queries triggered keywords)
  d. Device performance breakdown
  e. Geographic performance breakdown (top 20 locations by spend)
  f. Hour-of-day performance breakdown
  g. Previous audit results (to avoid repeating dismissed suggestions)

F3.3 — Gemini AI Prompt Design
  System prompt:
    "You are an expert Google Ads optimization AI for small brands.
     You analyze campaign data and return actionable, specific recommendations
     to reduce wasted spend and improve ROAS. You always return valid JSON.
     Focus on the highest-impact changes first. Be specific with numbers.
     Never recommend more than 15 total actions per audit."

  User prompt structure (send as structured JSON):
    {
      account: { currency, budget, goal },
      campaigns: [{ name, spend, revenue, roas, status, budget }],
      keywords: [
        { text, matchType, spend_7d, conversions_7d, cpc, quality_score }
      ],
      searchTerms: [
        { query, keyword_triggered, spend, clicks, conversions }
      ],
      devicePerformance: { mobile, desktop, tablet },
      previouslyDismissed: [recommendation types to avoid]
    }

  Expected Gemini response format (instruct model to return ONLY this):
    {
      healthScore: Number (0-100),
      summary: {
        wastedSpend: Number,
        projectedROASGain: Number,
        urgency: "high|medium|low"
      },
      recommendations: [
        {
          priority: "P1|P2|P3",
          type: "NEGATIVE_KEYWORD|BID_INCREASE|...",
          title: String,
          description: String,
          reasoning: String,
          targetEntity: { type, id, name },
          currentValue: Mixed,
          proposedValue: Mixed,
          projectedWeeklySavings: Number,
          projectedWeeklyRevenueGain: Number
        }
      ]
    }

F3.4 — Recommendation Execution
  When user clicks "Apply Now" for a recommendation:

  NEGATIVE_KEYWORD:
    → Call Google Ads API CampaignCriterionService.mutate
    → Add negative keywords to campaign or ad group
    → Update keyword aiTag in MongoDB

  BID_INCREASE / BID_DECREASE:
    → Call Google Ads API AdGroupCriterionService.mutate
    → Update keyword.bidAmount in MongoDB

  BUDGET_SHIFT:
    → Call Google Ads API CampaignBudgetService.mutate for both campaigns
    → Update campaigns.dailyBudget in MongoDB

  PAUSE_KEYWORD:
    → Call Google Ads API AdGroupCriterionService.mutate (status=PAUSED)
    → Update keyword.status in MongoDB

  AD_SCHEDULE:
    → Call Google Ads API CampaignCriterionService (ad schedules)

  On success: recommendation.status = 'applied', log to ActionLog
  On failure: recommendation.status = 'failed', log error, show user

F3.5 — Account Health Score Calculation
  Score = weighted average of 5 components (0-100 each):
    Keywords Health    (25% weight): % of keywords with conv rate > 0 + no waste
    Bid Strategy       (20% weight): appropriate strategy for account size/goal
    Budget Allocation  (20% weight): budget concentration on high-ROAS campaigns
    Ad Quality         (20% weight): average quality score across all keywords
    Waste Score        (15% weight): inverse of wasted spend percentage

  Score thresholds:
    80-100: Excellent (green)
    60-79:  Good (yellow)
    40-59:  Needs Work (orange)
    0-39:   Poor (red)

  Visual IF → THEN interface:
  IF: [metric] [operator] [value] [time window]
  AND/OR logic for multiple conditions
  THEN: [action] [value/type]
  WHEN: [schedule frequency] at [time]
  NOTIFY: [email/in-app toggles]

F4.2 — Available Metrics for Conditions
  - CPC (cost per click)
  - CPA (cost per acquisition)
  - ROAS (return on ad spend)
  - CTR (click-through rate)
  - Conversions (count)
  - Spend (total amount)
  - Impressions (count)
  - Quality Score (keyword level)
  - Budget Utilization (% of daily budget used)

F4.3 — Available Actions
  - Pause keyword / campaign / ad group
  - Enable keyword / campaign / ad group
  - Increase bid by X% or to ₹X
  - Decrease bid by X% or to ₹X
  - Increase daily budget by X% or to ₹X
  - Decrease daily budget by X% or to ₹X
  - Add keyword as negative
  - Send alert (no Google Ads change, just notify)

F4.4 — Pre-Built Rule Templates
  Template 1: "Pause zero-convert keywords"
    IF conversions (14d) = 0 AND spend (14d) > ₹500 AND CPC > ₹20
    THEN pause keyword
    SCHEDULE: Daily at 9:00 AM

  Template 2: "Boost high-ROAS ad groups"
    IF ROAS (7d) > 5 AND spend (7d) > ₹1,000
    THEN increase bid by 20%
    SCHEDULE: Daily at 9:00 AM

  Template 3: "Budget cap protection"
    IF budget utilization (1d) > 90%
    THEN send alert + decrease budget by 10%
    SCHEDULE: Hourly

  Template 4: "Night hours pause"
    IF hour = 23 OR hour = 0..6
    THEN pause all campaigns
    SCHEDULE: Hourly (checks time condition)

  Template 5: "Low quality score alert"
    IF quality score < 5 AND impressions (7d) > 100
    THEN send alert
    SCHEDULE: Daily at 9:00 AM

  Template 6: "Competitor bid spike detection"
    IF CPC (1d) > CPC (7d) * 1.5 AND spend (1d) > ₹200
    THEN send alert
    SCHEDULE: Daily at 9:00 AM

F4.5 — Rule Execution Engine (node-cron)
  Cron schedule overview:
    Hourly rules:   '0 * * * *'  (every hour)
    Daily rules:    '0 9 * * *'  (9 AM IST = 3:30 AM UTC)
    Weekly rules:   '0 9 * * 1'  (Monday 9 AM IST)

  Execution logic per rule:
    1. Fetch all active rules for schedule frequency
    2. For each rule, fetch relevant data from MongoDB (already synced)
    3. For each entity in scope (keyword/campaign/ad group):
       a. Evaluate all conditions
       b. Apply conditionLogic (AND/OR)
       c. If conditions met → execute action via Google Ads API
       d. Log result
    4. Save RuleExecutionLog document
    5. If notification enabled → create Notification document
    6. If email enabled → send via Resend

F4.6 — Rule Simulation
  Before saving a rule, user can click "Simulate"
  Backend evaluates conditions against current data (no changes made)
  Returns: how many entities would be affected and what action would be taken
  Frontend shows preview list to user

SECTION 13 — AUTHENTICATION & SECURITY
================================================================================

JWT STRATEGY:
  Access token:   15 minutes expiry, stored in httpOnly cookie + memory
  Refresh token:  7 days expiry (30 days if "remember me"), stored in httpOnly cookie
  Refresh token hash stored in MongoDB users.refreshToken field
  On refresh: old token invalidated, new one issued (rotation)

PASSWORD SECURITY:
  bcrypt with saltRounds = 12
  Minimum 8 characters, at least 1 number enforced at validation layer
  Password reset: time-limited token (1 hour), single-use, bcrypt hash stored

TOKEN ENCRYPTION (for Google Ads tokens):
  Algorithm: AES-256-GCM
  Key: derived from ENCRYPTION_KEY env var using crypto.scryptSync
  IV: random 16 bytes generated per encryption, stored alongside cipher text
  Implementation in /services/encryption.service.js

API SECURITY:
  helmet(): sets 11 HTTP security headers
  cors(): restricted to frontend domain only
  Rate limiting: 100 req/15min global, 10 req/15min on auth routes
  Input validation: Joi schemas on all POST/PUT/PATCH bodies
  MongoDB injection prevention: Mongoose sanitization
  XSS prevention: sanitize-html on any user-input stored to DB
  HTTPS only (enforced by Render.com and Vercel)

GOOGLE OAUTH SCOPES:
  Request minimal scopes needed:
  - openid, email, profile (for login)
  - https://www.googleapis.com/auth/adwords (for Google Ads)
  Store consent screen explanation in OAuth app settings

RULE EXECUTION FUNCTION (rules.service.js):

  async function executeRule(rule) {
    const account = await GoogleAccount.findById(rule.accountId);
    const tokens = decryptTokens(account);
    await refreshTokensIfNeeded(account, tokens);

    // fetch relevant entities based on scope
    const entities = await fetchEntitiesForScope(rule.scope, account);

    const matched = [];
    for (const entity of entities) {
      const meetsConditions = evaluateConditions(
        rule.conditions,
        rule.conditionLogic,
        entity
      );
      if (meetsConditions) matched.push(entity);
    }

    const results = [];
    for (const entity of matched) {
      const result = await executeAction(rule.action, entity, account, tokens);
      results.push(result);
    }

    await saveExecutionLog(rule, entities.length, matched.length, results);

    if (rule.notification.notifyOnFire && matched.length > 0) {
      await createNotification(rule, results);
      if (rule.notification.email) {
        await sendAlertEmail(rule, results);
      }
    }
  }

CONDITION EVALUATOR:
  function evaluateCondition(condition, entityMetrics) {
    const value = entityMetrics[condition.metric][condition.timeWindow];
    switch(condition.operator) {
      case 'gt':  return value > condition.value;
      case 'lt':  return value < condition.value;
      case 'gte': return value >= condition.value;
      case 'lte': return value <= condition.value;
      case 'eq':  return value === condition.value;
      case 'neq': return value !== condition.value;
    }
  }

  401: Unauthorized (invalid/expired token)
  403: Forbidden (insufficient plan or permissions)
  404: Not Found
  409: Conflict (e.g. duplicate email)
  429: Too Many Requests (rate limit)
  500: Internal Server Error
  503: Service Unavailable (Google Ads API down)

ERROR RESPONSE FORMAT:
  {
    "success": false,
    "error": {
      "code": "GOOGLE_ADS_TOKEN_EXPIRED",
      "message": "Your Google Ads connection has expired. Please reconnect.",
      "details": {}  // additional context (dev only in non-production)
    }
  }

ERROR CODES (custom):
  AUTH_001:         Invalid credentials
  AUTH_002:         Token expired
  AUTH_003:         Email not verified
  AUTH_004:         Account locked (5 failed login attempts)
  ACCOUNT_001:      Google Ads connection failed
  ACCOUNT_002:      Insufficient permissions
  ACCOUNT_003:      Token refresh failed
  ACCOUNT_004:      Account sync failed
  AUDIT_001:        Gemini AI unavailable
  AUDIT_002:        Insufficient data for audit (< 7 days of history)
  AUDIT_003:        Audit already running for this account
  RULES_001:        Rule condition configuration invalid
  RULES_002:        Rule action execution failed
  PLAN_001:         Free tier limit reached
  VALIDATION_001:   Request body validation failed

FRONTEND ERROR HANDLING:
  RTK Query: use isError + error states in components
  Axios interceptor: catch 401 → refresh token → retry request
  Global error boundary in React: catch unhandled errors → show fallback UI
  Toast messages for user-facing errors (non-blocking)
  Full-page error state only for critical failures (account disconnected, etc.)

GOOGLE ADS API ERRORS:
  INVALID_CUSTOMER_ID:   Show "Invalid Customer ID" message in UI
  CUSTOMER_NOT_FOUND:    Show reconnect prompt
  NOT_AUTHORIZED:        Show permissions explanation
  QUOTA_EXCEEDED:        Queue request for next day, notify user
  RESOURCE_EXHAUSTED:    Retry with exponential backoff (1s, 2s, 4s, 8s)
